Spring Boot

是一个简化Spring应用程序开发的框架，它可以让开发者快速构建和部署生产级的Spring应用

自动配置：Spring Boot基于一系列的条件判断自动配置Spring应用。例如，如果classpath下有spring-webmvc包，它会自动配置Spring MVC。

起步依赖 Starters：Spring Boot通过起步依赖来简化依赖管理。起步依赖是一组相关依赖的集合，它们被打包到一起，以便于开发者快速添加到项目中。例如，spring-boot-starter-web包含了构建Web应用所需的依赖。

无代码生成和XML配置 ：        Spring Boot不需要任何XML配置，也不需要额外的代码生成。大多数Spring Boot应用只需要很少的Spring配置。

内嵌服务器：Spring Boot提供了内嵌的Tomcat、Jetty或Undertow服务器，这使得开发者可以无需部署WAR文件到外部服务器即可运行应用。

命令行界面(LI）：Spring Boot CLI（命令行界面）是一个基于Groovy的命令行工具，可以用来快速构建Spring Boot应用原型。

Actuator:Spring Boot Actuator提供了生产环境中监控和管理应用程序的功能。它可以提供HTTP、JMX和SSH端点来查看应用程序的运行状态。

Spring Initializr:  这是一个在线工具，用于生成Spring Boot项目的初始结构。你可以选择项目的元数据（如项目名、描述、包名等），并选择所需的依赖，然后生成项目。

<dependencyManagement>
统一版本控制，该配置不会引入实际依赖，但会在子模块或当前模块中声明同 GAV（groupId、artifactId、version） 的 dependency 时强制使用指定版本，也就是说我们如果需要某个依赖，直接在我们的dependency中加入依赖就行，而无需指定版本。
 <type>pom</type>
 <scope>import</scope>
import作用域只能在<dependencyManagement>部分的<dependency>中使用
它只能用于type为pom的依赖
被导入的POM中的依赖管理配置会被合并到当前POM中

<parent>
引入标准父POM，从根本上改变了传统Maven项目中版本碎片化、配置冗余严重的局面。当一个项目声明 <parent> 引用该POM时，即自动继承了一整套经过严格验证的依赖版本组合、插件配置和构建属性，极大提升了开发效率与项目稳定性。

<dependencies>
项目中实际用到的依赖，如果已经在dependencyManagement（包括父pom中的dependencyManagement）定义了版本号，就无需指定版本号，从而使用集中定义的版本

<modules>
在 Maven 的多模块项目中，<modules> 标签用于定义当前项目包含的子模块。这允许你将一个大型的项目分解为多个更小、更易于管理的模块，同时保持它们之间的关系和依赖。
通过这种方式，你可以在父项目中统一管理依赖版本、插件配置等，而子模块可以继承这些配置，从而简化了项目的管理。同时，你可以在父项目目录下运行 Maven 命令，这些命令会自动应用到所有子模块上。
例如，运行 mvn clean install 会在父项目及其所有子模块上执行清理和安装操作。这对于构建、测试和部署整个项目非常有用。

二.springboot框架各层的关系和流程

1.仅仅有一个空的框架是没有用的，我们要给里面加些内容，这些内容一般分为四层，分别是controller层（控制层），service层（业务层），mapper层（数据持久层），model层（数据库实体层）。

2.controller层：它负责前后端交互，接受前端发送的请求（通过后端的路由地址发送），并调用service层，service层在返回数据给它，它在返回给前端。

service层：负责各种业务逻辑的处理，调用mapper层操作数据库，再对返回的数据进行各种业务上的处理，再返回给controll层。

mapper层：主要操作数据库，完成增删查改功能，把数据返回给service层。

model层：数据库的实体层，存放实体类，实现getset方法，是数据库的映射，变量的类型和名称必须和数据库相同，若名称由两个单词以上组成，第二个单词的首字母要大写。

3.springboot的流程

前端发送请求，controller层接受信息，然后调用service层的接口及其接口实现类，实现类在通过mapper层操作数据库，mapper层把数据返回给service层，然后在service层进行业务处理，然后在把数据返回给controller层。

理解spring mcv的过程
Servlet容器，如使用@WebServlet
mvc开发，如使用@GetMapping
Ioc容器中使用annotation配置，如使用@Component，@Autowired
Servlet容器中使用Ioc容器，如使用@Controller

1.定义web.xml配置Spring MVC提供的DispatcherServlet，这个web.xml是一个样板配置
2.有了这个配置，Servlet容器会首先初始化Spring MVC的DispatcherServlet，在DispatcherServlet启动时，它根据配置AppConfig创建了一个类型是WebApplicationContext的IoC容器，完成所有Bean的初始化，并将容器绑到ServletContext上。
3.因为DispatcherServlet持有IoC容器，能从IoC容器中获取所有@Controller的Bean，因此，DispatcherServlet接收到所有HTTP请求后，根据Controller方法配置的路径，就可以正确地把请求转发到指定方法，并根据返回的ModelAndView决定如何渲染页面。

spring中的事务传播
如果在一个声明了事务的方法中调用另一个同样声明了事务的方法，这样就涉及到事务传播，事务传播有多种级别，默认是REQUIRED，它的意思是，如果当前没有事务，就创建一个新事务，如果当前有事务，就加入到当前事务中执行，它能满足大部分需求。
事务传播的原理是spring把JDBC相关的Connection和TransactionStatus实例绑定到ThreadLocal。如果一个事务方法从ThreadLocal未取到事务，那么它会打开一个新的JDBC连接，同时开启一个新的事务，否则，它就直接使用从ThreadLocal获取的JDBC连接以及TransactionStatus。
可见事务能够正确传播的前提是方法调用在同一个线程。
如果要实现跨线程传播，就需要把Connection和TransactionStatus传递到新线程，实现起来非常复杂。

全自动orm和半自动orm
全自动orm,如Hibernate和JPA
对比Spring提供的JdbcTemplate，它和全自动ORM框架相比，主要有几点差别：

1.查询后需要手动提供Mapper实例以便把ResultSet的每一行变为Java对象；
2.增删改操作所需的参数列表，需要手动传入，即把User实例变为[user.id, user.name, user.email]这样的列表，比较麻烦。
 全自动ORM虽然自动实现了对象映射和对象传递，但它也有一些问题：
1.ORM框架总是引入了Attached/Detached状态，表示当前此Java Bean到底是在Session的范围内，还是脱离了Session变成了一个“游离”对象。很多初学者无法正确理解状态变化和事务边界，就会造成大量的PersistentObjectException异常。这种隐式状态使得普通Java Bean的生命周期变得复杂
2.Hibernate和JPA为了实现兼容多种数据库，它使用HQL或JPQL查询，经过一道转换，变成特定数据库的SQL，理论上这样可以做到无缝切换数据库，但这一层自动转换除了少许的性能开销外，给SQL级别的优化带来了麻烦。
3.ORM框架通常提供了缓存，并且还分为一级缓存和二级缓存，二级缓存是指跨Session的缓存，一般默认关闭，需要手动配置。二级缓存极大的增加了数据的不一致性，原因在于SQL非常灵活，常常会导致意外的更新。

但是JdbcTemplate的优势在于它的确定性：即每次读取操作一定是数据库操作而不是缓存，所执行的SQL是完全确定的，缺点就是代码比较繁琐，构造INSERT INTO users VALUES (?,?,?)更是复杂。

半自动ORM
所以，介于全自动ORM如Hibernate和手写全部如JdbcTemplate之间，还有一种半自动的ORM，它只负责把ResultSet自动映射到Java Bean，或者自动填充Java Bean参数，但仍需自己写出SQL。MyBatis就是这样一种半自动化ORM框架。
使用MyBatis最大的问题是所有SQL都需要全部手写，优点是执行的SQL就是我们自己写的SQL，对SQL进行优化非常简单，也可以编写任意复杂的SQL，或者使用数据库的特定语法，但切换数据库可能就不太容易。好消息是大部分项目并没有切换数据库的需求，完全可以针对某个数据库编写尽可能优化的SQL。
没有自动加载一对多或多对一关系的功能。

jmx和Spring Boot Actuator（执行器）
jmx是java EE内置的 一个程序，使用jconsole启动，可以在项目中创建MBeans然后通过jmx程序管理。
Spring Boot Actuator是spring boot内置的一个监控功能，内置了很多类型的监控信息也叫端点可供http、jmx、 ssh、telnet 等形式来监控和管理生产环境
默认情况下可以通过http访问，但默认只能访问health和info，其他需要配置才能访问，注意如果是非安全连接不能查看详情信息，配置了也无效。
Spring Boot > 2.2.x 默认禁用 JMX 暴露，需要添加以下设置才能使其工作：
spring.jmx.enabled=true，且默认只暴露health给jmx（以MBeans的形式）

当然，Spring Boot Actuator也支持自定义端点
